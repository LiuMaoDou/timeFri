# SQLite Session Storage Design

## Goal

Persist the single active timeFri session on the server so different browsers can continue the same timer and see the same ordered progress entries without authentication.

## Scope

- Store the active session and its progress entries in SQLite.
- Load the session once when the page opens and poll every three seconds afterward.
- Synchronize immediately after starting, recording progress, and ending a session.
- Keep theme preference in browser `localStorage`.
- Keep completed-session storage out of scope; Flomo remains the completed record destination.

## Architecture

The application will use `better-sqlite3` behind a focused repository module. The database file defaults to `data/timefri.sqlite` and can be overridden with `TIMEFRI_DATABASE_PATH`. The `data/` directory is ignored by Git and must be mounted as persistent storage in production.

The database contains one `active_session` row and ordered `session_entry` rows. SQLite transactions protect session creation, entry appends, and successful completion. A numeric version increases whenever the session changes so API clients can identify newer state.

## Data Model

`active_session`:

- `singleton_key`: fixed primary key value enforcing one active session
- `id`: UUID generated by the client
- `event_name`: trimmed event name
- `start_at`: Unix time in milliseconds
- `version`: monotonically increasing integer
- `updated_at`: Unix time in milliseconds

`session_entry`:

- `id`: integer primary key
- `session_id`: owning session UUID
- `position`: zero-based ordering value
- `content`: trimmed progress content

## API

`GET /api/session` returns the active session or `null`.

`POST /api/session` creates the active session. It returns `409` when another session already exists.

`PATCH /api/session` appends one progress entry in a transaction. It requires the session ID and rejects requests for a stale or missing session.

`POST /api/session/finish` reads the authoritative database state, optionally appends the final draft, sends the complete ordered content to Flomo, and deletes the session only after Flomo succeeds. A failed Flomo request leaves all data intact for retry.

## Client Data Flow

On mount, the page immediately fetches `/api/session`, then repeats the fetch every three seconds. The server response replaces client session state when it differs. Starting and recording progress wait for the API response and apply the returned authoritative session immediately.

Ending uses `/api/session/finish`; after success the client clears its state. Polling is paused while a mutation is in flight to avoid stale responses overwriting the mutation result.

The previous active-session `localStorage` key is removed after the first server synchronization. Theme storage remains unchanged.

## Concurrency And Errors

- A database singleton constraint prevents two simultaneous active sessions.
- Entry appends calculate the next position inside an immediate SQLite transaction.
- Mutations include the session ID so a browser cannot modify a replacement session accidentally.
- API validation retains the existing event-name and entry-length limits.
- Network and server failures are shown in the existing dialog/error UI and preserve server data.
- With no authentication, every visitor to the deployment shares and can modify the same active session. This is an explicit product constraint.

## Testing

- Repository tests cover initialization, singleton creation, ordered appends, session replacement protection, and deletion.
- Route tests cover validation, conflicts, missing sessions, Flomo failure retention, and successful deletion.
- Existing Flomo formatting and theme tests remain unchanged.
- Type checking, linting, production build, and browser QA verify the integrated client flow and polling behavior.

## Deployment

The production server must run a persistent Node.js process and mount the database directory on durable storage. Serverless filesystems without a persistent volume are unsupported by this SQLite version. A later PostgreSQL migration can preserve the API and replace only the repository implementation.
